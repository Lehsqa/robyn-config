# Admin Panel SQLAlchemy Parity Design

Date: 2026-02-28
Status: Approved
Decision: Option 3 (Hybrid shared core + ORM adapters)

## Context

The repository currently includes an early admin panel integration in `src/adminpanel` based on a separate source module in `qc_robyn_admin-main`. The scaffolded admin package and integration logic are Tortoise-centric, while generated backend projects support both ORMs (`tortoise`, `sqlalchemy`) across both design layouts (`ddd`, `mvc`).

Current blockers:
- `src/adminpanel/utils.py` hard-blocks non-Tortoise projects.
- Generated admin template wiring imports Tortoise runtime config (`TORTOISE_ORM`) directly.
- Core admin implementation is tightly coupled to Tortoise query/model APIs.

Goal: deliver a working admin panel for generated backend projects using either ORM provider without breaking existing admin API contracts.

## Goals

- Preserve current public admin API and route behavior:
  - `AdminSite`, `ModelAdmin`, `register(app)`
  - Existing route paths and HTML template contracts.
- Support both ORM providers with feature parity:
  - Auth/login/session flow
  - Model registry and permissions
  - List/detail CRUD
  - Pagination, filtering, searching
  - Inlines and import/export behavior
- Ensure SQLAlchemy projects provision admin tables through Alembic migrations generated by `adminpanel` command.
- Add integration coverage for all 4 generated project combinations:
  - `ddd/sqlalchemy`, `ddd/tortoise`, `mvc/sqlalchemy`, `mvc/tortoise`.

## Non-Goals

- Changing UI/UX of existing admin templates.
- Renaming exposed classes/functions or route paths.
- Introducing runtime auto-create table fallback for SQLAlchemy.

## Chosen Approach

### Option 3: Hybrid Shared Core + ORM Adapters (Recommended and Approved)

Keep shared admin workflows in core modules and move ORM-specific responsibilities behind adapter interfaces.

Benefits:
- Avoids full duplicate codebase split by ORM.
- Minimizes regression risk for existing Tortoise behavior.
- Enables parity fixes in one shared behavior layer.

## Architecture

### 1) ORM Adapter Contract

Refactor `src/adminpanel/template/orm/base.py` into a richer interface for:
- Model metadata introspection (fields, PK, relation info).
- Query operations (list/query/search/filter/order/paginate).
- Object operations (get/create/update/delete/batch delete).
- Relation and inline traversal helpers.
- Session/transaction lifecycle hooks needed by route handlers.

Implementations:
- `src/adminpanel/template/orm/tortoise.py` (compatibility adapter).
- `src/adminpanel/template/orm/sqlalchemy.py` (new adapter).

### 2) Shared Core on Contract

Refactor core modules to consume adapter APIs rather than ORM-native imports:
- `src/adminpanel/template/core/admin.py`
- `src/adminpanel/template/core/fields.py`
- `src/adminpanel/template/core/filters.py`
- `src/adminpanel/template/core/inline.py`
- `src/adminpanel/template/core/site.py`

Key invariants:
- Keep `AdminSite`/`ModelAdmin` signatures stable.
- Keep template context keys stable to avoid template regressions.

### 3) Template Wiring by ORM

Update `src/adminpanel/template/__init__.py.jinja2` to render ORM-specific initialization while keeping one `register(app)` entrypoint.

- Tortoise path remains compatible with current generated layout.
- SQLAlchemy path uses generated project SQLAlchemy engine/session components.

### 4) Generator Integration

Update `src/adminpanel/utils.py`:
- Allow both ORMs (remove hard Tortoise-only guard).
- Pass `orm` into template rendering context.
- Generate ORM-appropriate dependency and wiring updates.

## SQLAlchemy Migration Strategy

For SQLAlchemy projects, `adminpanel` command must generate Alembic migration revision(s) that create admin schema:
- `robyn_admin_users`
- `robyn_admin_roles`
- `robyn_admin_user_roles`
- Any additional permission-related tables required by current behavior parity.

Requirements:
- Migration files generated under existing project migration location:
  - DDD: `src/app/infrastructure/database/migrations/versions`
  - MVC: `src/app/models/sqlalchemy/migrations/versions`
- Correct `down_revision` resolution from latest existing revision.
- Full downgrade path included.

## Testing Strategy

### Unit tests

Add/extend tests to verify:
- `adminpanel` command supports both ORMs.
- ORM-specific template rendering and import wiring.
- SQLAlchemy migration file generation and revision linking logic.

### Integration tests

Create/extend integration tests to run matrix:
- `create` + `adminpanel` for all 4 combos.
- Validate generated files and ORM-specific wiring.
- Validate SQLAlchemy projects include generated admin Alembic revision.
- Smoke-run admin endpoints (`/admin/login`, model list route) for each combo.

### Regression guarantees

- Existing Tortoise combos continue to pass.
- SQLAlchemy combos become first-class and pass same smoke checks.

## Risks and Mitigations

- Risk: hidden Tortoise assumptions in core logic.
  - Mitigation: adapter-first refactor + strict integration matrix.
- Risk: SQLAlchemy relation/introspection mismatch for advanced fields/inlines.
  - Mitigation: explicit adapter metadata methods and targeted tests for inline/filter/search behavior.
- Risk: migration conflicts in projects with existing revisions.
  - Mitigation: deterministic `down_revision` detection and tests against seeded revision chains.

## Rollout

1. Land adapter contract + shared core refactor.
2. Add SQLAlchemy adapter and ORM-specific template wiring.
3. Add SQLAlchemy migration generation in `adminpanel` command.
4. Land full matrix integration tests.
5. Validate locally with full test suite and targeted generated-project smoke checks.

## Acceptance Criteria

- `robyn-config adminpanel <project>` succeeds for generated projects using both ORMs and both design layouts.
- Generated app boots and admin auth/list routes respond for all 4 combinations.
- SQLAlchemy projects receive valid Alembic migration(s) for admin tables.
- Public admin API and route/template contracts remain backward compatible.
